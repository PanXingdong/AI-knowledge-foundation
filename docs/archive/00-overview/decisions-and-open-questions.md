# 07 决策与待确认问题

## 1. 当前已定的方向

### 1.1 产品定位

已定：

```text
工程知识助手双入口产品：面向研发团队和研发 Agent 的工程知识供给层。
```

不是：

- 普通 PDF 问答
- 图谱展示大屏
- 单纯 RAG
- 单纯 Agent memory
- IDE 插件
- MCP 工具服务

### 1.2 部署位置

已定：

```text
核心知识服务部署在部门内网服务器，Agent 可同步只读本地知识包或调用本地 CLI/HTTP。
```

原因：

- 统一文档和知识版本。
- 统一权限。
- 统一审计。
- 群机器人和多个 Agent 共用同一套知识。
- 避免每个人本地维护完整知识库。

### 1.3 调用方式

已定：

```text
群机器人入口 + Agent 本地入口
```

群机器人服务团队成员日常提问。

Agent 本地入口优先使用 CLI、本地 HTTP、SDK 或只读知识包。

当前成品路线不走 MCP；已有 MCP 代码只作为早期实验遗留。

### 1.4 核心输出

已定：

```text
Context Pack
```

不是直接把 PDF、Markdown 或 chunk 原样丢给 Agent。

### 1.5 技术路线

已定：

```text
第一版：结构化文档 + Hybrid Retrieval + Context Pack + 群机器人 demo + Agent 本地 CLI/HTTP demo
后续：Knowledge Graph + 人工审核 + 版本失效 + 影响分析
```

第一版先验证群机器人问答是否比人工翻文档更快、更可信，以及 Agent 本地入口拿 Context Pack 是否比直接给文件更稳定。图谱负责关系和影响分析，但只有实验结果证明需要时才进入后续增强。

方向冻结和阶段门见：

```text
docs/18-方向冻结与阶段门.md
```

## 2. 必须尽快确认的问题

### 2.1 第一批样本范围

已定：

```text
混合工程文档样本
```

第一批优先覆盖：

```text
高通供应商文档
博世供应商文档
内部 SPEC
技术架构文档
详细设计文档
测试设计或问题复盘
```

建议选择标准：

- 文档数量适中。
- 业务痛点明显。
- 有文档 owner、领域确认人或评分人可确认。
- 有接口、约束、测试项。
- 能验证供应商约束、内部 SPEC、设计和测试资料之间的上下文组装价值。
- Agent 使用频率高。

### 2.2 第一批文档

需要确认：

- 高通、博世、内部 SPEC、架构、详细设计、测试资料分别放在哪里。
- 文档总量多少。
- PDF 是否有扫描件。
- 是否包含 Word/Markdown/HTML。
- 是否有版本号。
- 是否涉密。
- 是否允许调用外部 LLM。

当前状态：

```text
指定工作目录下尚未发现真实 PDF/Word 样本文档。
```

需要把 10 份真实样本文档登记到 `samples/README.md`。

### 2.3 权限边界

需要确认：

- 哪些 Agent 可以查。
- 哪些项目文档可以查。
- 哪些人可以上传文档。
- 哪些人可以审核实体/关系。
- 是否需要按密级隔离。

### 2.4 审核责任人

需要确认：

- 文档 owner 是谁。
- 领域确认人是谁。
- 任务评分人是谁。
- 谁确认抽取结果。
- 谁确认版本失效。
- 审核 SLA 是否需要定义。

### 2.5 第一个验收场景

必须从真实问题里选。

推荐二选一：

```text
场景 A：Agent 查询某个供应商接口或内部 SPEC 相关改动前必须遵守的约束和测试点。
场景 B：输入高通/博世等供应商文档要求，系统输出与内部 SPEC、架构设计和测试资料相关的上下文。
```

### 2.6 模型和数据合规

需要确认：

- 是否可以使用外部商业 LLM。
- 是否必须内网私有化模型。
- 文档是否可以出内网。
- embedding 是否可使用外部服务。
- 日志里是否能保存查询内容。

## 4. 需求边界检查

如果后续讨论偏离，可以用这些问题拉回：

1. 这个能力是否让 Agent 更容易使用部门资料？
2. 是否比直接给 Agent 文件更稳定？
3. 是否能追溯到文档证据？
4. 是否能处理版本和失效？
5. 是否能被多个 Agent 统一调用？
6. 是否能在第一版真实场景里验证？

如果答案是否定的，就不放入阶段 1。
