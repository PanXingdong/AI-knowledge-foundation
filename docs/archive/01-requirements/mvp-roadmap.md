# 06 MVP 路线

## 1. 路线调整

当前路线从“直接做完整知识图谱”调整为：

```text
阶段 0：直接给 Agent 文件，建立基线
阶段 1：结构化文档 + 检索 + Context Pack + 群机器人 demo + Agent 本地入口 demo
阶段 2：知识图谱 + 人工审核 + 版本失效 + 影响分析
```

第一阶段的目标是证明“群机器人给人用、Agent 本地入口给 Agent 用”这两个成品入口是否值得继续投入，而不是一次性完成终局平台。

## 2. 第一批样本范围

第一批样本范围：

```text
混合工程文档样本
```

选择原因：

1. 真实目标是让 Agent 使用部门资料，不是只解决某一个模块。
2. 供应商文档和内部文档都有 PDF/Word/HTML、表格、版本、约束和术语问题。
3. Agent 直接读大文档时容易漏掉跨文档约束、内部 SPEC 对齐关系和证据来源。
4. 高通、博世、内部 SPEC、技术架构、详细设计和测试资料能覆盖更真实的部门知识供给场景。

如果第一轮样本过散，实验任务必须按一个真实工作流收敛，例如“供应商约束如何落到内部 SPEC/架构/测试”。

## 3. 第一批 10 份样本文档

真实文件当前尚未放入 `.`。第一阶段要求从业务目录复制或登记 10 份真实文档到 `samples/raw/`，并更新 [samples/README.md](../samples/README.md)。

样本类型建议覆盖：

1. 高通平台/BSP/接口约束文档。
2. 博世供应商接口、诊断、通信或集成文档。
3. 其他供应商平台、协议或工具链文档。
4. 内部需求/SPEC 文档。
5. 内部技术架构文档。
6. 内部详细设计文档。
7. 内部接口控制或配置说明。
8. 内部测试设计、测试报告或问题复盘。
9. 含重要表格的 PDF/Word 文档。
10. 多栏、扫描件或 OCR 风险文档。

## 4. 阶段 0：基线测试

目标：

```text
验证“直接给 Agent 文件”到底差在哪里。
```

做法：

- 选择 3-5 个真实 Agent 任务。
- 让 Agent 直接读取原始 PDF 或由用户提供文档。
- 记录准确率、遗漏率、证据引用、token、耗时和人工修正次数。

输出：

- 基线测试记录。
- 失败案例清单。
- 是否值得做 Context Pack 的初步判断。

## 5. 阶段 1：两周轻量实验

目标：

```text
验证“结构化文档 + 检索 + Context Pack”是否能支撑群机器人问答和 Agent 本地调用。
```

范围：

- 10 份混合工程样本文档。
- Docling / MinerU / Unstructured 解析对比。
- 结构化 blocks。
- 全文检索。
- 向量检索。
- Context Pack 组装。
- 群机器人问答 demo。
- Agent 本地 CLI 或本地 HTTP demo。

暂不做：

- Domain Graph。
- 完整人工审核台。
- 版本失效推理。
- 复杂影响分析。

## 6. 阶段 1 验收标准

必须满足：

1. 明确第一批样本范围：混合工程文档样本。
2. 登记 10 份真实样本文档。
3. 完成直接给 Agent 文件的基线测试定义。
4. 完成 Context Pack 实验方案。
5. 完成解析器对比指标：页码、span、表格、阅读顺序、OCR。
6. 完成 Agent 使用效果指标：准确率、遗漏率、token、耗时、证据正确率。
7. 完成群机器人问答效果指标：回答准确率、证据正确率、人工翻文档节省时间。
8. 文档明确图谱、审核、版本管理是后续增强，不是第一版前置依赖；MCP 不进入当前成品路线。

## 7. 阶段 2 进入条件

只有当阶段 1 至少满足以下条件，才进入阶段 2：

- Context Pack 相比直接给文件，关键任务准确率提升明显。
- 关键约束遗漏率下降。
- token 或耗时有明显下降。
- 证据引用正确率可接受。
- 群机器人回答比人工翻文档更快，并能稳定附证据。
- Agent 本地入口能稳定返回 Context Pack 和 evidence trace。
- 出现单纯检索难以解决的跨文档多跳关系需求。

阶段 2 再建设：

- 知识图谱。
- 人工审核。
- 版本失效。
- 影响分析。
- 实体卡片。
