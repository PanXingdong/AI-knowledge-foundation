# 01 需求定义

## 1. 项目定位

本项目暂命名为 **Agent Knowledge Hub**。

它是一个部署在部门内部的工程知识产品，用来把已有文档和资料加工成团队成员和 AI Agent 都能直接使用的上下文能力。

一句话定义：

```text
把部门沉淀在 PDF、SPEC、设计、测试、供应商资料里的工程知识，变成人能在群里问、Agent 能在本地调用的共享知识底座。
```

## 2. 背景问题

当前直接把文档给 Agent 或让 Agent 自己读文件，会遇到这些问题：

1. 文档太多、太大，无法稳定塞进上下文。
2. Agent 每次重新读文档，成本高，结果不一致。
3. PDF 里的章节、表格、页码、图和跨页内容容易丢失。
4. 没有统一的版本适用范围，旧资料可能污染新任务。
5. 没有人工确认机制，AI 抽取出的关系没人敢直接信。
6. 没有跨文档组合能力，难以做影响分析。
7. 不同 Agent 各自检索，知识使用方式不统一。
8. 无法统计哪些知识被用过、错过、过期或需要补充。

## 3. 目标用户

第一版重点服务两个入口，而不是先做完整知识门户或图谱大屏：

主要用户：

- 在群里提问工程文档问题的研发工程师、测试工程师、架构师和模块 owner
- Codex / Copilot / Cursor / Claude / 自研研发 Agent
- 使用这些 Agent 的研发工程师
- 维护知识的架构师、模块 owner、文档 owner、测试负责人

## 4. 核心使用场景

### 4.1 Agent 写代码前获取上下文

Agent 修改某个模块前，调用系统获取：

- 相关文档章节
- 接口定义
- 模块约束
- 已确认的风险点
- 相关测试项
- 版本适用范围
- 原文证据

### 4.2 Agent 做缺陷分析

输入缺陷现象、日志、模块和项目，系统返回：

- 相关设计约束
- 相关接口和依赖
- 相关历史文档说明
- 可能影响范围
- 需要检查的测试或配置

### 4.3 Agent 做设计或评审

Agent 生成 SDD、review 代码或设计时，系统返回：

- 必须遵守的供应商限制
- 内部 SPEC 对应章节
- 需求到设计到测试的链路
- 版本变更差异

### 4.4 人工审核知识

专家确认 AI 抽取出的实体、关系、约束和版本范围，保证进入主图谱的知识可信。

### 4.5 群机器人问答

团队成员在企业微信、飞书或钉钉群里直接提问，系统返回：

- 简短结论
- 关键依据
- 文档、版本、页码和章节
- 相关约束或测试点
- 不确定项和需要人工确认的地方

## 5. 阶段划分

### 5.1 第一阶段：轻量实验

第一阶段目标不是建设完整知识图谱，而是验证：

```text
结构化文档 + 检索 + Context Pack 是否能同时支撑群机器人问答和 Agent 本地调用。
```

第一阶段要做：

- 选择一个领域和 10 份真实样本文档。
- 对比 Docling / MinerU / Unstructured 的解析质量。
- 生成结构化文档块和最小证据字段。
- 建立全文检索和向量检索。
- 组装 Context Pack。
- 做群机器人问答 demo。
- 做 Agent 本地 CLI 或本地 HTTP demo。
- 与“直接给 Agent 文件”做基线对比。

第一阶段不做：

- 完整 Domain Graph。
- 完整人工审核台。
- 复杂版本 diff 和失效推理。
- 大屏展示。
- 全领域覆盖。

### 5.2 第二阶段：完整知识底座

只有当第一阶段证明 Context Pack 明显优于直接给文件后，才进入第二阶段。

第二阶段再做：

- 实体/关系抽取。
- 人工审核。
- 知识图谱。
- 版本失效。
- 影响分析。
- 实体卡片。
- 团队级知识治理。

## 6. 目标系统范围

目标系统最终要做：

- 文档上传或同步
- PDF/Word/Markdown/HTML 解析
- 文档结构化：文档、页、章节、块、表格、证据 span
- 基础检索：全文检索、向量检索
- 基础图谱：实体、关系、证据绑定
- 人工确认：实体和关系审核
- 版本管理：文档版本、关系适用范围、失效标记
- 人工入口：群机器人问答
- Agent 调用：本地 CLI、本地 HTTP、SDK 或只读知识包
- 输出 Agent 可消费的 Context Pack

目标系统也暂不做：

- 完整 IDE 插件
- 自动替 Agent 改代码
- 无审核自动入库
- 全公司通用知识库
- 复杂知识图谱大屏
- 完整缺陷系统替代品

## 7. 与普通 RAG 的区别

普通 RAG：

```text
PDF -> chunk -> embedding -> 相似度召回 -> 塞给 LLM
```

本项目：

```text
文档 -> 结构化 -> 证据绑定 -> 实体/关系抽取 -> 人工确认
     -> 版本管理 -> 混合检索 -> Context Pack -> 群机器人 / Agent 本地调用
```

RAG 是其中的检索手段，不是整个系统。

## 8. 与“直接把文件给 Agent”的区别

直接给文件：

- Agent 自己找重点。
- 每次都重新读。
- 容易漏掉跨文档关系。
- 无法保证版本正确。
- 没有人工确认。
- 返回格式不稳定。

本系统：

- 预先解析和结构化。
- 一次加工，多次复用。
- 支持跨文档组合。
- 每条结论有证据链。
- 有审核和版本失效。
- 返回结构化 Context Pack。
- 群机器人和 Agent 本地入口复用同一套证据。

## 9. 成功标准

第一阶段成功不以“图谱多漂亮”为标准，而以人和 Agent 是否能稳定使用为标准：

1. 群机器人能回答真实工程文档问题，并给出证据来源。
2. Agent 能通过本地入口拿到当前任务相关上下文。
3. 返回内容能追溯到文档版本、章节、页码和原文 span。
4. 相比直接给文件，准确率、遗漏率、token、耗时至少有一个维度明显改善，且不能明显牺牲证据正确率。
5. 解析器对比能明确选出第一阶段默认解析链路。
6. 图谱、审核、版本失效是否进入二阶段，由实验结果决定。
