# 08 两周轻量实验设计

## 1. 实验目标

验证以下判断是否成立：

```text
把工程 PDF 解析成结构化文档块，并通过混合检索组装 Context Pack，
能让 Agent 比直接读取原始文件更快、更准、更少遗漏。
```

本实验不验证完整知识图谱，不建设完整审核台，不做复杂版本失效。

## 2. 第一批样本范围

第一轮实验样本范围：

```text
混合工程文档样本
```

选择依据：

- 覆盖高通、博世等供应商资料，以及内部 SPEC、技术架构、详细设计和测试资料。
- 能验证 PDF/Word/HTML 解析、页码、span、表格、阅读顺序和 OCR。
- 能验证 Agent 是否能把供应商限制、内部规范和设计约束放到同一个上下文里。
- 更贴近最终目标：给多个研发 Agent 提供统一的工程文档知识底座。

## 3. 10 份样本文档

真实样本文档当前未放入目录。实验开始前，必须把 10 份文档登记到：

```text
.\samples\README.md
```

样本文档类型：

| 编号 | 类型 | 用途 | 状态 |
|---|---|---|---|
| 1 | 高通平台/BSP/接口约束文档 | 验证供应商接口、平台限制和版本约束 | 待提供 |
| 2 | 博世供应商接口/诊断/通信文档 | 验证供应商规范、表格和术语检索 | 待提供 |
| 3 | 其他供应商平台/协议/工具链文档 | 验证跨供应商资料接入 | 待提供 |
| 4 | 内部需求/SPEC 文档 | 验证内部规范和供应商文档关联 | 待提供 |
| 5 | 内部技术架构文档 | 验证架构约束和模块上下文 | 待提供 |
| 6 | 内部详细设计文档 | 验证实现级上下文和接口说明 | 待提供 |
| 7 | 内部接口控制/配置说明 | 验证接口、配置项和约束检索 | 待提供 |
| 8 | 内部测试设计/测试报告 | 验证测试关注点和覆盖要求 | 待提供 |
| 9 | 问题复盘/缺陷分析文档 | 验证历史风险和证据回溯 | 待提供 |
| 10 | 扫描件、多栏或表格密集文档 | 验证 OCR、阅读顺序和表格结构 | 待提供 |

## 4. 实验任务

至少选择 3 个真实 Agent 任务：

### 任务 A：查约束

```text
某个供应商接口或内部 SPEC 相关改动前，需要注意哪些约束？
```

### 任务 B：查接口/机制说明

```text
某个供应商接口、协议机制或内部接口应该如何使用，有哪些限制？
```

### 任务 C：生成测试关注点

```text
基于某个供应商约束或内部设计变更，生成需要关注的测试点。
```

如果业务侧已有更高频任务，以真实任务替换上述任务。

## 5. 对照组设计

### 5.1 基线组：直接给 Agent 文件

Agent 获取原始 PDF 或由用户提供文件路径，让 Agent 自己读取和判断。

记录：

- 是否找到正确章节。
- 是否遗漏关键约束。
- 是否引用错误文档或页码。
- token 使用量。
- 完成耗时。
- 人工修正次数。

### 5.2 实验组：Context Pack

系统先解析文档并建立检索索引。Agent 调用：

```text
get_context_pack(project, module, task_type, query)
```

Context Pack 返回：

- 相关摘要。
- 关键约束。
- 相关文档片段。
- 文档名、版本、页码、章节、span_id。
- 置信度。
- 开放问题。

## 6. 两周计划

### 第 1-2 天：样本和任务准备

- 选择 10 份能代表供应商资料和内部工程文档的混合样本。
- 收集 10 份真实文档。
- 定义 3-5 个真实 Agent 任务。
- 建立基线测试表。

### 第 3-5 天：解析器对比

- 用 Docling / MinerU / Unstructured 解析同一批文档。
- 按页码、span、表格、阅读顺序、OCR 评分。
- 选出第一阶段默认解析链路。

### 第 6-8 天：检索和 Context Pack

- 生成结构化 blocks。
- 建全文索引和向量索引。
- 实现 Context Pack 组装规则。
- 准备 REST API 或 MCP demo。

### 第 9-10 天：Agent 对比实验

- 同一批任务分别跑基线组和实验组。
- 记录准确率、遗漏率、token、耗时、证据正确率。
- 汇总结论。

## 7. 决策输出

实验结束必须回答：

1. Context Pack 是否明显优于直接给文件？
2. 哪个解析器适合作为第一阶段默认方案？
3. 是否需要进入知识图谱阶段？
4. 是否需要人工审核台？
5. 是否需要版本失效机制？

如果答案不明确，继续优化轻量方案，不进入完整平台建设。
