# 12 样本文档与场景选择依据

本文用于回答三个方向性问题：

1. 第一阶段为什么不再只选 QNX 单领域？
2. 10 份样本文档应该怎么选？
3. 真实 Agent 任务应该怎么选？

## 1. 第一阶段样本选择标准

候选样本不是按技术兴趣选，而是按能否验证 Agent Knowledge Hub 的真实价值选。

评分标准：

| 维度 | 说明 | 权重 |
|---|---|---|
| Agent 使用频率 | 这类文档是否经常被 Agent 用于编码、分析、评审、测试 | 高 |
| 文档痛点 | 是否存在大 PDF、跨文档、表格、多版本、术语复杂问题 | 高 |
| 约束密度 | 文档里是否有大量限制条件、接口规则、适配注意事项 | 高 |
| 跨文档价值 | 是否需要把供应商文档、内部 SPEC、设计和测试资料放到同一上下文里 | 高 |
| 可验证性 | 是否能设计 3-5 个真实任务并有标准答案 | 高 |
| 确认人支持 | 是否有人能确认答案、约束和证据是否正确 | 高 |
| 后续扩展价值 | 是否能自然扩展到图谱、版本失效、影响分析 | 中 |

## 2. 当前默认选择

第一轮实验默认选择：

```text
混合工程文档样本
```

也就是从以下资料中选 10 份真实文档：

- 高通平台、BSP、接口或工具链文档。
- 博世供应商接口、诊断、通信或集成文档。
- 其他供应商平台、协议或工具链文档。
- 内部需求、SPEC、接口控制文档。
- 内部技术架构和详细设计文档。
- 内部测试设计、测试报告、问题复盘文档。

理由：

1. 系统目标是让 Agent 使用部门资料，不是只回答某个模块的问题。
2. 供应商文档和内部文档的组合，最能验证 Context Pack 是否比直接给文件更稳定。
3. 真实任务常常不是“查一份 PDF”，而是“把供应商约束、内部 SPEC、设计方案和测试要求对齐”。
4. 混合样本能覆盖 PDF 解析、Word/HTML 接入、表格、多栏、OCR、版本和证据链。

## 3. 不再把 QNX 作为唯一默认领域

QNX 适配仍然可以作为样本来源之一，但不再作为第一阶段唯一领域。

如果 QNX 文档在第一批资料中高频、典型、可验证，可以纳入样本；如果高通、博世、内部 SPEC 或架构文档更能代表真实 Agent 使用场景，应优先选择这些资料。

判断标准不是“哪个领域技术上更完整”，而是：

```text
哪些文档最能证明 Agent Knowledge Hub 是否值得做。
```

## 4. 10 份样本文档选择依据

10 份文档不是随便凑数，要覆盖解析和 Agent 使用的主要风险。

必须尽量覆盖：

| 类型 | 覆盖目的 |
|---|---|
| 供应商平台 PDF | 验证长 PDF、章节、页码、术语 |
| 供应商接口/机制 PDF | 验证接口、限制条件、表格 |
| 内部 SPEC | 验证内部约束和供应商文档关联 |
| 内部架构设计 | 验证模块上下文和设计约束 |
| 内部详细设计 | 验证实现级上下文 |
| 测试设计/问题复盘 | 验证测试关注点和历史风险 |
| 扫描件或 OCR 风险文档 | 验证 OCR 能力 |
| 多栏或表格密集文档 | 验证阅读顺序和表格结构 |

建议第一批槽位：

| 编号 | 样本类型 | 用途 |
|---|---|---|
| 1 | 高通平台/BSP/接口约束文档 | 验证供应商接口限制和版本约束 |
| 2 | 博世供应商接口/诊断/通信文档 | 验证供应商规范和表格术语 |
| 3 | 其他供应商平台/协议/工具链文档 | 验证跨供应商资料接入 |
| 4 | 内部需求/SPEC 文档 | 验证内部规范和供应商约束对齐 |
| 5 | 内部技术架构文档 | 验证架构上下文 |
| 6 | 内部详细设计文档 | 验证实现级上下文 |
| 7 | 内部接口控制/配置说明 | 验证接口、信号、配置项检索 |
| 8 | 内部测试设计/测试报告 | 验证测试关注点 |
| 9 | 问题复盘/缺陷分析文档 | 验证历史风险和证据回溯 |
| 10 | 扫描件、多栏或表格密集文档 | 验证解析器边界 |

## 5. 真实 Agent 任务选择依据

任务必须来自真实工作流，不要为了实验硬造。

优先选择：

1. 过去一个月真实问过 Agent 的问题。
2. 新人经常查文档才能回答的问题。
3. 容易漏供应商约束或内部 SPEC 对齐关系的问题。
4. 需要引用文档证据的问题。
5. 能由文档 owner、领域确认人或评分人给出标准答案的问题。

第一阶段至少需要 3 类任务：

```text
查约束
查接口/机制
生成测试关注点
```

推荐任务形态：

| 任务类型 | 示例 |
|---|---|
| 查约束 | 某个供应商接口或内部 SPEC 相关改动前，需要注意哪些限制和前置条件？ |
| 查接口/机制 | 某个供应商接口、协议机制或内部接口应该如何使用，有哪些限制？ |
| 生成测试关注点 | 基于供应商约束或内部设计变更，生成需要覆盖的测试点。 |
| 一致性检查 | 高通/博世文档里的要求，在内部 SPEC 或架构文档中是否有体现？ |
| 设计辅助 | Agent 写设计方案前，需要从哪些文档提取约束、接口和测试要求？ |

## 7. 进入实验的最低条件

必须满足：

1. 至少 10 份样本文档登记完成。
2. 至少覆盖 1 份供应商文档、1 份内部 SPEC/需求文档、1 份设计文档、1 份测试或问题复盘文档。
3. 至少覆盖 1 份含表格文档、1 份多栏文档、1 份扫描件或 OCR 风险文档。
4. 至少 3 个真实任务登记完成。
5. 每个任务都有确认人或评分人。
6. 每个任务都有标准答案要点。
7. 每个任务至少能指出 1 个预期证据来源。

不满足这些条件时，不跑实验，只继续补样本和任务定义。
