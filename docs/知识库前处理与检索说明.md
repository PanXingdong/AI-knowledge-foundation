# AI Knowledge Foundation 前处理与检索说明文档

> 适用范围：本仓库 `src/agent_knowledge_hub/` 下的 Python 包，以及配套的 `scripts/`（PowerShell 包装器）、`schemas/`、`samples/`。
> 目标读者：需要理解 / 维护 / 扩展「工程文档 → 可溯源 Context Pack」流水线的工程师。
> 性质：本文是面向实现的中文说明，覆盖 **Layer1（解析入库）+ Layer2（检索与 Context Pack）** 的端到端机制；权威契约仍以 `schemas/layer1.processed.v1/` 与 `src/agent_knowledge_hub/contract.py` 为准。

---

## 0. 一分钟速览

把**异构工程文档**（PDF / DOCX / Markdown / HTML / TXT）统一加工成「**规范文档模型（canonical document）+ 分块（chunks）+ 证据跨度（evidence spans）**」，再在其上做**本地、零外部服务**的混合检索，产出一份带**证据回溯（evidence trace）**的 **Context Pack**，供人与 Agent 直接当作上下文使用。

```
documents
  -> inventory / manifest        清点与采样清单
  -> parsing                     解析为块（含 PDF 文本质量评估 + OCR 兜底）
  -> canonical document model    规范模型（document / version / section / block / evidence）
  -> chunks + evidence spans     按章节切块，块携带证据 id
  -> parse quality gate          解析质量门（决定是否进 Context Pack）
  -> retrieval                   FTS5 + 本地稀疏向量 + 词法/主题混合打分
  -> Context Pack                Markdown / JSON 上下文包
  -> evidence trace              按 evidence_id 回溯到原文 block/页码/章节
```

整套流程被切成 **Layer1** 与 **Layer2** 两段，中间用一份**版本化契约**衔接（见 §3）。

---

## 1. 两层边界（最重要的概念）

| | Layer1（解析入库） | Layer2（检索与消费） |
|---|---|---|
| 输入 | 原始文档文件 | Layer1 的处理产物（`canonical-document.json` + `chunks.jsonl`） |
| 产物 | 规范模型 + 分块 + 质量报告 | 索引 + Context Pack + 证据回溯 |
| 核心模块 | `inventory` `parsers` `builder` `chunker` `pipeline` `incremental` | `contract` `fts_index` `vector_index` `retrieval` `layer2_run` |
| 契约 | **写出** `layer1.processed.v1` | **校验并消费** `layer1.processed.v1` |

> 关键设计：Layer2 **不直接碰原始文件**，只消费 Layer1 的产物目录（`processed_dir`）。这让「解析」与「检索」可以独立演进、独立验收，也让协作方只要产出符合契约的 `canonical-document.json + chunks.jsonl` 就能接入 Layer2。

`README.md` 把这条边界画成：

```
canonical-document.json + chunks.jsonl
  -> contract validation -> FTS + local vector indexes
  -> retrieval -> Context Pack -> evidence trace
  -> group bot / local agent entry
```

> 注：仓库里有历史 MCP 代码（`mcp_server.py`），但 **MCP 不是当前产品路线**，仅作历史实验保留。

│   document / document_version / sections / blocks / evidence    │

      canonical-document.json
      chunks.jsonl

## 3. Layer1 → Layer2 契约（`layer1.processed.v1`）

这是两层之间唯一的硬约定。Schema 文件在 `schemas/layer1.processed.v1/`，运行时校验在 `contract.py`。

**目录形态**：每个文档版本一个目录，里面恰好两份文件：

```
processed/<title-slug>/<document_version_id>/
  canonical-document.json   # 规范模型，一份
  chunks.jsonl              # 切片，一行一个 chunk
```

**`canonical-document.json` 顶层结构**（对应 `models.CanonicalDocument`）：

| 字段 | 含义 |
|------|------|
| `schema_version` | 必须等于 `layer1.processed.v1`（`models.CANONICAL_DOCUMENT_SCHEMA_VERSION`） |
| `document` | 逻辑文档：`document_id / title / source_type / owner / project / supplier / created_at` |
| `document_version` | 版本：`document_version_id / document_id / version / file_path / file_hash / created_at` |
| `sections` | 章节树：每个含 `section_path`（如 `["1","2"]`）、`title`、`page_start/page_end` |
| `blocks` | 内容块：`block_type`(heading/paragraph/table)、`text`、`section_path`、`order`、`page_*` |
| `evidence_spans` | 证据跨度：`evidence_id`、`block_id`、`section_path`、`page`、`text`、`text_hash`（每个 block 一条） |
| `parse_report` | 解析报告：parser 名、格式、页数、各类计数、`warnings`、`quality_report` |

**`chunks.jsonl` 每行结构**（对应 `models.Chunk`）：
`chunk_id / document_version_id / section_path / page_start / page_end / text / evidence_ids / embedding_id / metadata`。

**`contract.validate_processed_dir()` 校验什么（节选）**：

- `schema_version` 存在且等于 `layer1.processed.v1`，否则 `unsupported_schema_version`。
- `document / document_version / parse_report` 的必填字段齐全。
- `document_version.document_id` 必须等于 `document.document_id`。
- **引用完整性**（这是契约的核心价值）：
  - 每个 `block.section_path` 必须指向已声明的 section；
  - 每个 `evidence.block_id` 必须指向已声明的 block；
  - 每个 `chunk.evidence_ids` 必须指向已声明的 evidence span；
  - block / evidence / chunk 的 `document_version_id` 必须与版本一致。
- `quality_report.status` 必填、`score` 应为数值（缺失记 warning）。
- `chunks.jsonl` 必须存在且至少一行。

> 校验产物：`processed-contract-validation.{json,md}`，`is_valid` 决定 Layer2 是否继续。

## 4. Layer1 详解

### 4.1 文档清点（`inventory.py`）

`build_document_inventory()` 在一个或多个根目录下发现可入库文档，生成清单与**采样清单 CSV**（manifest），作为后续入库的输入。

- **遍历**：`_iter_candidate_files()` 深度遍历，跳过 `DEFAULT_EXCLUDED_DIR_NAMES`（`.git`、`__pycache__`、`node_modules`、`.venv`、`AppData`、`Windows`、`Program Files` 等）。
- **过滤**：只收 `SUPPORTED_EXTENSIONS`（`.pdf/.docx/.md/.markdown/.html/.htm/.txt`）；支持 `include/exclude` 关键字；`max_files`（默认 200）、`max_file_mb`（默认 100MB）限流。
- **去重**：默认按 `content_hash`（SHA-256）去重，同内容只留一份。
- **元数据推断**：从路径推断 `supplier`（Qualcomm / Bosch / QNX / GB…）与 `source_type`（spec / architecture / review checklist / supplier document…）。
- **产物**：`document-inventory.json`、`document-inventory.md`、`raw-docs-sample-manifest.csv`（列：`sample_id,file_path,document_title,slot_type,owner,project,supplier,document_version,content_hash`）。

> manifest 的 `slot_type` 列对应入库时的 `source_type`；`file_path` 为占位符（`待提供` 等，见 `utils.PLACEHOLDER_VALUES`）的行会在入库时被跳过。

### 4.2 解析（`parsers.py`）

`parse_document(path)` 按扩展名分派到 5 个解析器，统一产出 `ParsedDocument(source_format, parser_name, page_count, blocks, warnings, quality_report)`。块类型只有三种：`heading` / `paragraph` / `table`。

| 格式 | 解析器 | 关键行为 |
|------|--------|----------|
| `.md/.markdown` | `stdlib-markdown-block-parser` | 正则识别 `#` 标题（带 level）、`\|...\|` 表格、空行分段 |
| `.txt` | `stdlib-text-block-parser` | 仅按空行分段为 paragraph |
| `.html/.htm` | `stdlib-html-parser` | `HTMLParser` 提取 `h1-h6/p/li/表格`，忽略 `script/style` |
| `.docx` | `python-docx` | 段落样式 `Heading N` → heading，表格转 `a \| b` 文本 |
| `.pdf` | `pypdf`（必要时 `pypdf+rapidocr`） | 见下 §4.3 |

> 解析依赖是**可选**的：`pypdf`（PDF）、`python-docx`（DOCX）、`pymupdf+rapidocr+onnxruntime`（OCR 兜底）。缺失时抛 `DocumentParseError` 并说明要装什么。`dependency-check` 命令可预检。

### 4.3 ★ PDF 文本质量评估与 OCR 兜底（本流水线最有特色的一段）

PDF 是质量最不可控的来源。`parse_pdf()` 的策略是「**先信文本层，质量不行再上 OCR**」：

1. `_extract_pdf_text_pages_with_pypdf()` 用 pypdf 抽每页文本层。
2. `assess_pdf_text_quality()` 给全文打质量诊断，产出 `reason_codes`：
   - `text_too_short`：字符数 < 40（基本是扫描件/空文本层）；
   - `mojibake_suspect_ratio_high`：CJK 多但夹杂大量疑似乱码字符（按「每千 CJK 字符的乱码数 > 50」判定）；
   - `latin1_mojibake_ratio_high`：非 CJK 文本里 Latin-1 乱码字符占比 > 0.12（编码错乱）。
3. 一旦有 `reason_codes`，`should_use_ocr=True`，尝试 `_parse_pdf_with_rapidocr()`：用 PyMuPDF 把每页渲染成 2x 位图，喂 RapidOCR（`text_score≥0.45` 过滤低置信行）。
4. 三种结局都会被如实记录在 `parser_name` 和 `warnings` 里：
   - OCR 成功 → `parser_name="pypdf+rapidocr"`，质量报告 `status="recovered_by_fallback"`；
   - OCR 不可用/失败/无文本 → 退回 pypdf 文本，`status="low_quality"`，并打 `pdf_text_quality_low_ocr_*` 警告；
   - 本就合格 → `parser_name="pypdf"`，`status="ok"`。

**质量分（0–100）** 由 `PdfTextQualityReport.to_dict()` 计算：从 100 起，按命中的 reason 扣分（太短 −45、乱码按比例扣等），OCR 兜底成功 +18。这个分数后续会喂给**质量门**（§4.6）。

> 设计取舍：质量评估是**启发式**而非真识别——它的目的不是"读懂"PDF，而是**在入库前就把不可信的文本拦下来**，避免乱码/空文本污染检索。非 PDF 走 `_build_generic_quality_report()`（按是否有正文块、字符数判定 `ok/low_quality`）。

### 4.4 规范模型构建（`builder.py`）

`build_canonical_document()` 把 `ParsedDocument` 的扁平块列表，提升为带**章节树**和**证据**的规范模型。

- **ID 方案**：全部用 `utils.stable_id(prefix, *parts)` = `prefix_` + `sha256(parts)[:16]`，内容确定即 id 确定（可复现、可去重）。前缀：`doc / docver / sec / blk / span / chunk / chunkmerge`。
- **章节树有两种模式**：
  - **PDF 书签模式**（有 outline 时）：把 PDF 书签拍平成 `页码 → section_path` 映射作为主结构；`["0"]` 段兜底封面/目录等书签前内容；解析器在书签内识别到的标题再生成细分子章节（计数器在跨书签时重置）。同名书签重复出现会**合并并延长 page_end**，保证后续块仍能引用到覆盖它的 section。
  - **标题模式**（Markdown / DOCX / HTML）：按 heading 的 level 维护多级计数器，`section_path` 形如 `["1","2","1"]`。
- **证据跨度**：**每个 block 生成一条 `EvidenceSpan`**，记录 `page / section_path / block_id / text / text_hash`。这是后续 evidence trace 能回溯到原文的根基。
- **parse_report** 汇总 parser 名、页数、section/block/table 计数、`has_page_numbers`、warnings、quality_report。

### 4.5 分块（`chunker.py`）

`build_chunks()` 是**章节感知（section-aware）+ 重叠（overlap）**的切块：

| 维度 | 取值 / 行为 |
|------|------------|
| 切分单位 | block（不切碎句子，以解析块为最小单位累积） |
| 目标大小 | `max_chunk_chars`，默认 **1600 字符** |
| 重叠 | `overlap_chars`，默认 **160 字符**（保留末尾若干 block 作为下一块开头） |
| 切块时机 | 当前块非空且（**章节路径变化** 或 累计长度将超过 `max_chunk_chars`）时 flush |
| 章节变化 | flush 时**不保留 overlap**（跨章节不重叠，避免污染）；超长 flush 才保留 overlap |
| 定位 | chunk 携带 `section_path / page_start / page_end / evidence_ids` |
| 兜底 | `overlap >= max` 时自动压到 `max//10`；`max < 50` 直接报错 |

> 与旧 QNX 工具按「字符/行」滑窗不同，这里以**解析块为粒度**累积，并在**章节边界强制切断**，让每个 chunk 尽量落在同一节内、语义更聚焦；`evidence_ids` 让 chunk 能反查到来源 block。

### 4.6 解析质量门（`quality.py` / `_context_pack_gate`）

并非所有解析结果都配进入检索。质量门 `_context_pack_gate()` 判定 `allowed_for_context_pack`：

```
allowed = (quality_status ∈ {ok, recovered_by_fallback})
          AND (quality_score 为空 或 ≥ 40.0)
          AND chunks.jsonl 存在
```

任一不满足则记 `gate_reasons`（如 `quality_status_low_quality`、`quality_score_below_40`、`missing_chunks_jsonl`）。`build_parse_quality_summary()` 汇总成 `parse-quality-summary.{json,md}`，列出每份文档的状态、分数、是否放行、失败输入（来自 `ingest-summary.json` 的 `failed`）。

> 注意：质量门是**软门**——Layer2 检索时，被门挡下的 chunk 默认不参与；但若它被 FTS/向量命中，会**临时放行**（gate bypass，见 §5.3），避免漏掉强相关内容。

### 4.7 入库编排与增量（`pipeline.py` / `incremental.py`）

- `ingest_file()`：单文件 → 解析 → 建模 → 切块 → 写 `canonical-document.json` + `chunks.jsonl`，返回 `IngestResult`。
- `ingest_manifest()`：读 manifest CSV，逐行入库；占位/缺失路径跳过，解析异常记入 `failed`；写 `ingest-summary.json`。
- `ingest_manifest_incremental()`：维护 `ingest-state.json`（按文件绝对路径记 `content_hash`）。**哈希未变则跳过**（`status=unchanged`），变了才重跑，输出 `ingest-run-summary.json`（含 `unchanged/changed/processed/failed` 计数），同时回写一份兼容老格式的 `ingest-summary.json`。

> 供应商推断 `_infer_supplier()`：manifest 显式列优先，否则从 `slot_type+title` 文本里匹配「高通/Qualcomm → Qualcomm」「博世/Bosch → Bosch」。

## 5. Layer2 详解

### 5.1 两套索引

Layer2 在 Layer1 产物上建两套**互补**索引，都从 `_iter_latest_processed_versions()` 取——即**每个文档只取最新版本**（按 `created_at` / 文件 mtime / version_id 排序）。

**① SQLite FTS5（`fts_index.py`）— 词法精确召回**

- `build_fts_index()` 建虚表 `fts_chunks`，正文列 `document_title / section_titles / chunk_text` 参与索引，其余列 `UNINDEXED` 仅作回带。
- 查询 `query_fts_index()`：`_build_fts_match_query()` 把查询切成 token，每个加 `*` 前缀通配并 `AND` 连接（只保留含 ASCII 字母/数字/下划线的 token，长度≥2），用 `bm25()` 排序。
- 产物：`chunks.fts.sqlite` + `.summary.json`。

**② 本地稀疏向量（`vector_index.py`）— 无依赖语义近似**

- 嵌入策略 `local-hashed-token-v1`，**纯本地、零模型**：`_build_sparse_vector()` 把文本拆成
  - ASCII token（`ascii:foo`，下划线再拆子词）、
  - **CJK 2–4 元组 n-gram**（`cjk:出境`、`cjk:出境审`…，解决中文无空格分词问题）、
  - 同义词组特征（`CJK_SYNONYM_GROUPS`，如 `出境/跨境/境外/海外`、`审批/评估/审核/批准`）。
- 检索时按 `document_frequency` 算 IDF 加权，做**余弦相似度**排序。
- 产物：`chunks.vector.json`（含全量稀疏向量 + DF）+ `.summary.json`。

> 设计取舍：FTS5 擅长精确词/英文符号；本地向量靠 n-gram + 同义词组补"中文近义/换词"的召回。两者都**离线可跑**，不引入任何向量服务或 embedding 模型。

### 5.2 检索：加载与邻接合并（`retrieval._load_processed_chunks`）

- 只加载**最新版本**文档的 chunk，并把每个文档级质量门信息（`quality_status / score / allowed / gate_reasons / warnings`）贴到 chunk 上。
- 文本会经 `_trim_trailing_appendix_noise()` 去掉尾部附录噪声。
- **邻接合并块**：`_build_neighbor_merged_chunks()` 用滑动窗口（2–7 个相邻 chunk，合并后 ≤2800 字符）额外生成「合并块」，并合并它们的 `section_titles` 与 `evidence_ids`。这让**答案恰好跨在两个 chunk 边缘**时仍能被作为一个整体候选召回（id 前缀 `chunkmerge`）。混合了"噪声块+正文块"的窗口会被跳过。

### 5.3 检索：混合打分（`_build_candidate_scores`）

每个候选 chunk 的 `overall_score` 是多路信号之和：

```
overall_score =
    词法分  _score_tokens(query 全量 token, 命中标题/正文加权)
  + BM25    _score_bm25(本地 BM25 上下文)
  + FTS 加成   fts_bonus      （来自 SQLite FTS5 命中）
  + 向量加成   vector_bonus   （来自本地稀疏向量命中）
  + 连贯性  _coherence_bonus  （句子完整度）
  + 结构    _structure_bonus  （列表/小标题等结构信号）
  + 证据质量 _evidence_quality_adjustment
  + 附录方法查询 _appendix_method_query_bonus（查"附录/方法"类问题时偏置）
  + 质量门调整 _quality_gate_adjustment（低质量文档轻微降权）
  + 主题对齐 _topic_alignment_bonus（architecture/backend/api/governance/rollout）
  + 稀疏块惩罚 _thin_chunk_penalty（过短/信息稀薄降权）
```

其中：
- **clause hits**：查询被 `_split_query_clauses()` 切成子句，逐句计分，命中阈值（≥2.0 或短句>0）记录 `clause_hits`，用于多子句覆盖度排序。
- **retrieval_signals**：`_derive_retrieval_signals()` 标注每个 chunk 的命中来源（词法 / bm25 / fts / 向量 / 主题），写进 Context Pack 便于解释为什么选它。
- **gate bypass**：`build_context_pack_for_processed_dir()` 里，被质量门挡下但**被 FTS 或向量命中**的 chunk 会被加回候选池（`external_gate_bypass_chunks`），避免强相关内容因解析质量分被误杀。

### 5.4 检索：选取（`_select_candidates`）

在排序后的候选上做约束选取：

- `top_k`（默认 8）：最终条数上限。
- `per_document_limit`（默认 2）：每个文档最多入选几条，保证**多文档覆盖**而非被单一长文档刷屏。
- **主题/子面覆盖**：依据查询推导的 `desired_topics` 与 `requested_subfacets`，在选取时尽量覆盖不同子主题（`_pick_topic_candidate`、缺失子面时的软放宽 `_candidate_needs_soft_document_limit_exception`）。
- **去重**：`_is_overlapping_duplicate()` 用序列/词重叠比剔除高度重复的 chunk。
- **元数据过滤**：`metadata_filters` 支持 `source_type / project / supplier / document_version`（`SUPPORTED_METADATA_FILTERS`）。

### 5.5 产物：Context Pack 与证据回溯

- `ContextPackResult`（`schema_version="context-pack.v1"`）→ `write_context_pack_bundle()` 写出：
  - `context_pack.md`：可直接喂大模型的 Markdown 上下文（按章节聚合、带来源标注）；
  - `context_pack.json`：结构化全量（含每个 chunk 的 text、score、evidence_ids、retrieval_signals）；
  - `context_pack-summary.json`：精简摘要（不含全文，便于审阅/落库）。
- `trace_evidence_in_processed_dir(evidence_id)`：按 `evidence_id` 回到原文，返回 `EvidenceTraceResult`——文档标题/版本、`source_path`、页码、`section_path/section_titles`、原文 `text`、以及引用了该证据的 chunk 列表。**这是"每条结论都能点回原文"的实现。**
- `compare_context_pack_against_reference()`：把自动 Context Pack 与人工参考 Markdown 比，产出**覆盖/缺失项**的 gap report（`gap-report` 命令），用于评测召回质量。

### 5.6 一键验收：`layer2-run`（`layer2_run.py`）

`run_layer2_acceptance()` 把 Layer2 串成一条**可验收**的流水线，并产出 readiness 结论：

```
1. validate_processed_dir            契约校验（不过则直接列 blocker）
2. build_fts_index + build_vector_index
3. build_context_pack_for_processed_dir
4. 取 Context Pack 第一个 evidence_id → trace_evidence_in_processed_dir
5. 汇总 is_ready / blockers → layer2-run-summary.{json,md}
```

`blockers` 任一存在（契约非法、索引失败、Context Pack 为空、证据回溯失败等）则 `is_ready=False`；CLI 传 `--require-ready` 时据此返回非零退出码，可直接用于 CI / 验收门禁。

## 6. 如何运行（CLI：`python -m agent_knowledge_hub.cli <command>`）

> Windows 下另有 `scripts/*.ps1` 包装器（如 `ingest-documents.ps1`、`start-context-pack-api.ps1`），内部仍是调用这些命令。运行前先 `pip install -r requirements.txt`，PDF/OCR 另装 `requirements-ocr.txt`；`PYTHONPATH` 指向 `src`。

```powershell
# 0) 预检解析/OCR 依赖
python -m agent_knowledge_hub.cli dependency-check

# 1) 清点文档、生成采样清单
python -m agent_knowledge_hub.cli inventory `
  --root-dir "D:\some\docs" --output-dir ".\data\inventory" --sample-size 50

# 2-A) 按清单入库（全量）
python -m agent_knowledge_hub.cli manifest `
  --manifest-path ".\data\inventory\raw-docs-sample-manifest.csv" `
  --out-dir ".\data\processed"
# 2-B) 增量入库（跳过未变文件）
python -m agent_knowledge_hub.cli manifest --incremental `
  --manifest-path "...csv" --out-dir ".\data\processed"
# 2-C) 单文件入库
python -m agent_knowledge_hub.cli file --file-path ".\samples\raw\README.md" `
  --out-dir ".\data\processed" --title "Sample" --source-type sample

# 3) 校验契约
python -m agent_knowledge_hub.cli validate-processed `
  --processed-dir ".\data\processed" --require-valid

# 4) 质量汇总
python -m agent_knowledge_hub.cli parse-quality-summary `
  --processed-dir ".\data\processed" --output-dir ".\data\quality"

# 4-A) 阶段1 Observe 质量诊断（可选 --policy-path）
python -m agent_knowledge_hub.cli evaluate-quality `
  --processed-dir ".\data\processed" `
  --output-dir ".\data\quality-observe"

# 5) legacy 直接建索引（仅用于迁移兼容）
python -m agent_knowledge_hub.cli build-fts-index    --processed-dir ".\data\processed" --index-path ".\data\indexes\chunks.fts.sqlite"
python -m agent_knowledge_hub.cli build-vector-index --processed-dir ".\data\processed" --index-path ".\data\indexes\chunks.vector.json"

# 5-A) 阶段0：构建并显式激活 release
python -m agent_knowledge_hub.cli build-release `
  --processed-dir ".\data\processed" `
  --releases-dir ".\data\releases"

python -m agent_knowledge_hub.cli activate-release `
  --manifest-path ".\data\releases\<release_id>\release-manifest.json" `
  --active-pointer ".\data\releases\active-release.json"

# 6) 使用 ready manifest 绑定的索引生成 release-aware Context Pack
python -m agent_knowledge_hub.cli context-pack `
  --processed-dir ".\data\processed" --query "这个文档说明了什么？" `
  --top-k 5 --per-document-limit 2 `
  --release-manifest-path ".\data\releases\<release_id>\release-manifest.json" `
  --fts-index-path ".\data\releases\<release_id>\indexes\chunks.fts.sqlite" `
  --vector-index-path ".\data\releases\<release_id>\indexes\chunks.vector.json" `
  --output-dir ".\data\context-pack"

# 7) 一键 Layer2 验收（契约+索引+Context Pack+证据回溯）
python -m agent_knowledge_hub.cli layer2-run `
  --processed-dir ".\samples\golden" --output-dir ".\agent-artifacts\run" `
  --query "What constraints should the agent use?" --require-ready

# 8) 按指定 release 回溯历史证据
python -m agent_knowledge_hub.cli trace `
  --processed-dir ".\data\processed" `
  --evidence-id "<evidence_id>" `
  --release-manifest-path ".\data\releases\<release_id>\release-manifest.json"
```

`evaluate-quality` 只生成 Observe 报告和 publication/quarantine preview；即使报告给出 `quarantine`、`block_document` 或 `block_release` 等 hard recommended action，诊断成功仍返回 exit 0。该命令绝不修改 release、索引、active pointer、processed 输入或检索行为。preview 中的 `would_exclude_*` 仅表示“若未来启用门禁将排除什么”，不会在当前阶段实际排除任何文档或 chunk。`parse-quality-summary` 在迁移期间继续保持可用和兼容；Candidate enforcement 属于后续 PR，本阶段不实现 release v2 或 production 门禁。

`build-release` 只构建 release-bound FTS、向量索引和质量 baseline，并把 manifest 推进到 `ready`；它**绝不自动激活**。必须单独运行 `activate-release`，才会原子写入 active pointer。release-aware `context-pack` 必须传 `--release-manifest-path`，并使用该 ready manifest 中 `indexes.fts.path` 与 `indexes.vector.path` 绑定的索引。上面第 5 步的 `data/indexes` 直接索引示例仅用于迁移兼容，不是 release-aware 发布路径；不传 manifest 时保留原有 legacy 行为。

`candidate` manifest 可用于发布前/影子验证：不传索引时执行 lexical 检索，也可显式传入 release ID 匹配的临时候选索引。`ready` manifest 是生产语义：不传 FTS/向量路径时自动解析 manifest 绑定路径；显式传入时必须与规范路径完全相同。每次检索都会重验文档、索引哈希和 release ID；BGE `.npz` 同时校验矩阵与 `.npz.metadata.json`。外部复制即使伪装成相同 release ID 也不会被 ready 检索接受。

`trace` 不传 `--release-manifest-path` 时保持 legacy 最新版本行为；传入后只在该 release 固定的文档版本内回溯，因此后续入库不会改变历史证据结果。

阶段0只建立 release 身份、产物一致性和质量 baseline，不预设质量阈值或硬发布门禁；阶段1才会基于真实 QNX、Qualcomm 和正常 PDF 语料的生产 baseline 增加硬门禁。

其余命令：`gap-report`（召回缺口）、`prepare-eval-run` / `record-eval-output` / `score-eval-run` / `check-eval-business-readiness` / `eval-run-status`（评测 A/B harness，见 §7）。

## 7. 评测 harness（`eval_setup.py`，简述）

为回答"Context Pack 到底有没有用"，仓库内置一套 **baseline（直接喂原始文件）vs context_pack（喂 Context Pack）** 的人工 A/B 评测流程：`prepare-eval-run` 生成配对 prompt 与打分占位 → 真实 Agent 跑出输出用 `record-eval-output` 回填 → checker 用 `record-eval-review-decision` 记裁决 → `score-eval-run` / `check-eval-business-readiness` 判定该轮是否可作为"业务证据"。模板在 `experiments/templates/`。

---

## 8. 对外服务（`service.py`）

`service.py` 用 FastAPI 暴露核心能力（Context Pack、检索、证据回溯等），由 `scripts/start-context-pack-api.ps1` 启动。**约定**（见 `README.md` 开发规则）：群机器人 / 本地 Agent 适配层必须是**薄入口**，调用 Core API，**不得自己重写检索逻辑**，且每个 Context Pack 条目都要保留证据回溯。

## 9. 已知限制与可改进点

| 项 | 现状 | 备注 / 方向 |
|----|------|------------|
| PDF 结构还原 | pypdf 文本层 + OCR 兜底，无版面/表格语义重建 | 需要正文级表格/标题结构时可评估 Docling/Marker（依赖重、需模型） |
| 质量评估是启发式 | mojibake/latin1/字符数阈值 | 拦的是"明显坏"的文本，非语义级质量 |
| 中文分词 | FTS5 默认分词对中文有限，靠向量侧 n-gram 补 | 中文为主场景可考虑 trigram tokenizer / 分词预切 |
| 向量是本地哈希稀疏向量 | 零依赖、可离线，但语义能力弱于真 embedding | 需要更强语义时可接入真 embedding（但会引入模型/服务依赖，与"离线"取舍） |
| 检索打分含较多启发式 | `retrieval.py` 含大量主题/附录/结构 bonus | 规则多、可解释但需随样本调参；signals 已外露便于诊断 |
| 产物体积 | 向量索引为全量 JSON | 大库时 JSON 向量索引会偏大，可改持久化存储 |
| MCP | 历史实验代码 | 非当前路线，勿当作支持入口 |

## 10. 一句话总结

> 这套系统把**异构工程文档**经 Layer1（解析 + PDF 质量门 + OCR 兜底 + 章节感知分块 + 证据跨度）加工成符合 `layer1.processed.v1` 契约的规范产物，再由 Layer2（FTS5 + 本地稀疏向量混合检索）产出**带证据回溯的 Context Pack**——全程零外部服务、可离线、可一键验收（`layer2-run`），作为人与 Agent 共用的可溯源上下文底座。
