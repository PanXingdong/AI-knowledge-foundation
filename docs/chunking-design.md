# 分块设计说明

> 适用范围：`src/agent_knowledge_hub/chunker.py`
> 目标读者：需要理解、维护或调整分块行为的工程师

---

## 1. 设计目标

这套分块不是为了存储，而是为了**被检索后直接当作上下文使用**。最终产物是喂给 Agent 或大模型的 Context Pack，因此分块的核心约束是：

- 不能太长（超出检索模型的有效编码窗口）
- 不能太短（信息稀薄，召回后无实际价值）
- 内容要语义聚焦（不能把不相关的章节混在一起）

---

## 2. 核心参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_tokens` | 512 | 每块 Token 上限，优先于字符限制 |
| `max_chunk_chars` | 1600 | 字符上限，仅当 `max_tokens=0` 时生效 |
| `overlap_chars` | 160 | 相邻 chunk 的重叠字符数 |
| `min_chunk_chars` | 10 | 低于此长度的 chunk 直接丢弃 |

512 token 的上限匹配 BGE-M3 等主流检索模型的有效编码窗口。

---

## 3. 分块规则

### 3.1 章节边界强制切断

```
章节路径（section_path）变化 → 立即 flush，且不保留 overlap
```

工程文档的章节之间往往是不同主题（如第 3 章讲接口定义，第 4 章讲错误码）。跨章节合并会导致检索时命中不属于该主题的内容，降低精度。章节切换时不保留 overlap，是因为跨章节的重叠内容反而会引入语义噪声。

### 3.2 标题与正文合并

```
heading 块 → 缓存到 pending，等下一个 paragraph/table 到来时合并
```

孤立的标题行对检索没有价值（"3.2.1 信号定义" 单独成一个 chunk 无法被任何有意义的查询命中）。把标题合并进后续内容块，让每个 chunk 自带"我属于哪个主题"的上下文。

如果标题后紧跟表格，标题也会合并进表格的 chunk，而不是单独成块。

### 3.3 表格原子化

```
table 块 → 独立 chunk，不拆分，不跨块
```

表格的一行如果被切断就失去含义。技术规范里的表格（信号列表、接口参数、错误码）往往是最有价值的参考信息，必须作为整体被召回。遇到表格时，先把前面已积累的段落 flush 掉，再让表格独占一个 chunk。

### 3.4 Overlap 保留

```
普通段落超出 Token 预算 → flush，保留末尾 overlap_chars 字符作为下一块开头
```

答案有时恰好落在两个 chunk 的交界处。160 字符的末尾重叠让相邻 chunk 各自都能覆盖这段内容，避免"缝隙漏检"。

### 3.5 段落内句子级拆分

当单个段落块本身就超出 Token 预算时，按以下优先级拆分：

1. 空行（`\n\n`）— 段落边界，最安全
2. CJK 句末标点（`。！？`）— 始终安全
3. 英文句末（`. ` + 大写字母）— 排除缩写词（`e.g.`、`fig.`、`v7.1` 等）

拆出的子块会重新走累积逻辑，多个短句可以合并进同一个 chunk，不会每句话单独成块。

---

## 4. 噪声过滤

分块前先过滤两类无检索价值的内容，不让其进入索引。

### 4.1 噪声章节

| 过滤条件 | 说明 |
|----------|------|
| `section_path[0]` 为 `"0"` | 封面、版权页等书签前内容 |
| `section_path[0]` 为 `"Contents"` | 目录页 |
| `section_path[0]` 为 `"Index"` | 索引页 |
| 单字母章节 + 含字母导航栏 | "A ¦ B ¦ C…" 形式的字母索引分组 |

单字母章节（如附录 "A"）只有在块内容包含字母导航栏时才过滤，正常的附录章节不受影响。

### 4.2 碎片块

短于 120 字符、且匹配以下模式的块直接跳过：

- 裸页码（`12`、`- 5 -`、`Page 12`）
- 日期（`2026-05-08`、`8 May 2026`）
- 版本号（`v7.1`、`Revision 3`）
- 版权/机密声明（`Confidential`、`© QNX`）
- 文件编号格式（`80-82727-100 CW`）

---

## 5. CJK 感知 Token 估算

这套系统处理中英混合的工程文档。固定字符数上限会导致问题：
- 纯中文文档：1600 字符 ≈ 1600 token，大幅超出模型窗口
- 纯英文文档：1600 字符 ≈ 400 token，利用率只有一半

因此使用轻量估算公式，不引入外部 tokenizer 依赖：

```
token 数 ≈ CJK字符数 + 非CJK字符数 ÷ 4
```

CJK 字符（中/日/韩）在 BGE-M3、mBERT 等多语言模型中约 1 字符对应 1 token；ASCII/拉丁字符约 4 字符对应 1 token。

## 6. chunk_id 生成

```python
chunk_id = "chunk_" + sha256(document_version_id + "\n" + text)[:16]
```

**内容寻址**：相同 `document_version_id` + `text` 永远生成相同 ID，跨重新构建保持稳定，不会触发下游的误删误建。

**版本隔离**：`document_version_id` 参与哈希，同一段文本在不同文档版本中得到不同 ID。文档更新后新版 chunk 自然获得新 ID，旧索引可以精确清理，无需全量重建。

---

## 7. 每条规则背后的取舍

| 规则 | 如果不这样做 |
|------|-------------|
| 章节边界切断 | 跨章节 chunk 被召回后，Agent 拿到的是混合了多个主题的噪声内容 |
| 标题合并进正文 | 标题单独成块，被召回后 Agent 只看到一行标题，无法据此回答问题 |
| 表格原子化 | 表格被切断，召回的半张表没有意义，甚至误导 Agent |
| Overlap 保留 | 跨 chunk 边界的答案永远无法被完整召回 |
| 噪声过滤 | 页码、日期、版权声明会占用 top-k 配额，挤掉真正有价值的内容 |
| CJK Token 估算 | 中文文档的 chunk 实际超出模型窗口，embedding 质量下降 |
| 内容寻址 ID | 重新入库后 ID 变化，下游向量索引需要全量重建 |

## 8. Word 文档（.docx）支持

当前已支持，解析器基于 `python-docx`（需单独安装）。

**解析规则：**
- `Heading N` 样式段落 → `heading` 块
- 普通段落 → `paragraph` 块
- 表格 → 每行转为 `单元格1 | 单元格2` 格式，整表作为 `table` 块

**限制：** DOCX 无页码信息，chunk 的 `page_start/page_end` 为空，证据回溯无法定位到具体页。

**入库方式：**
```powershell
# 单文件
python -m agent_knowledge_hub.cli file --file-path "doc.docx" --out-dir ".\data\processed" --title "标题" --source-type spec

# 依赖预检
python -m agent_knowledge_hub.cli dependency-check
```

## 9. 与检索层的关系

chunk 携带的字段直接服务于 Layer2 检索：

| 字段 | 用途 |
|------|------|
| `section_path` | FTS 索引的 `section_titles` 列，标题命中加权 |
| `page_start / page_end` | 证据回溯（trace）定位到原文页码 |
| `evidence_ids` | 通过 `evidence_id` 回溯到原始 block 的具体文本 |
| `metadata.source_type` | 支持元数据过滤（`source_type / project / supplier`） |

邻接合并块（`chunkmerge_*`）由检索层在运行时动态生成，不写入 `chunks.jsonl`，用于覆盖恰好跨在两个原始 chunk 边界的答案。
