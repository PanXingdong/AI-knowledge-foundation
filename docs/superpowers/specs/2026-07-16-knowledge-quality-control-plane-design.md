# 知识质量控制平面设计

日期：2026-07-16
状态：待用户审阅
适用仓库：AI Knowledge Foundation

## 1. 背景

当前系统已经具备工程文档解析、PDF 文本质量检测、OCR 兜底、规范文档模型、章节感知切块、证据跨度、混合检索和增量入库等能力。这些能力证明了端到端链路可行，但还不足以保证新增知识在进入生产前已经达到可信标准。

现有主要问题包括：

- PDF 质量检测主要关注文本长度和乱码，无法系统识别多栏错序、表格破坏、标题碎片、页眉页脚污染和局部页面失败。
- PDF 解析策略主要在整份文档级别选择，无法对文本页、扫描页和复杂表格页分别路由。
- 现有质量门属于软门，低质量内容被检索命中后仍可能绕过门禁进入 Context Pack。
- `canonical-document.json`、`chunks.jsonl`、BM25/FTS 和向量索引没有统一绑定到不可变 release，存在主数据和索引版本不一致的风险。
- 当前证据可定位到 block 和页码，但缺少页面坐标、原文字符范围、解析配置版本和发布批次等完整来源信息。
- 解析器升级或新增质量规则后，缺少自动影响分析和批量重处理机制。
- QNX 等独立前处理流程仍包含摘要替换、字符滑窗和单独 SQLite 入库路径，容易形成厂商特例和知识质量分叉。
- 质量问题主要在回归或用户问答阶段暴露，发现时间晚于知识发布。

本设计不以“针对某份 QNX 或 Qualcomm 文档增加清洗规则”为核心。目标是建立一套与厂商无关、能够自动发现、阻断、恢复和持续治理知识质量问题的控制平面。

## 2. 目标

### 2.1 核心目标

建立以下闭环：

```text
原始文档
  -> 文档画像
  -> 逐页解析路由
  -> 多维质量评估
  -> 自动恢复或隔离
  -> 结构化切块
  -> 候选发布验收
  -> 原子发布
  -> 线上监控
  -> 影响分析与重处理
```

系统完成后应具备：

- 新文档在发布前自动发现明显解析和结构缺陷。
- 低质量内容默认不能进入生产检索。
- 每条已发布知识都能追踪到原始文件版本、页码和原文区域。
- 所有检索索引和主数据来自同一个 release。
- 解析器、质量规则或源文件变化后，只重处理受影响内容。
- 用户反馈用于补充未知盲区，而不是承担主要质量发现责任。
- QNX、Qualcomm 等文档只作为复杂样本，不成为代码中的厂商分支。

### 2.2 成功定义

本项目的成功不是“保证所有 PDF 完美解析”，而是：

- 坏知识更难进入生产。
- 发布前可以发现已知类别的严重缺陷。
- 无法自动修复的内容会被明确隔离。
- 问题可以定位、复现、比较和批量修复。
- 发布可以安全回滚。
- 质量提升可以通过稳定指标证明。

具体百分比阈值在建立阶段 0 基线后确定，设计阶段不凭经验虚构目标值。

## 3. 非目标

首轮不包括：

- 按文件名、厂商或产品维护大量专用清洗分支。
- 对所有 PDF 无条件运行完整 OCR 或重型版面模型。
- 让 LLM 单独决定内容是否可以发布。
- 自动发布证据不可追踪或质量不确定的内容。
- 建设复杂的运营管理后台。
- 一次性替换全部现有解析、检索和存储实现。
- 在知识质量控制平面中解决所有回答生成层幻觉；回答层仍需保留独立 guardrail。

## 4. 设计原则

### 4.1 原始数据不可变

原始文件按内容哈希保存和识别。解析、切块和索引均为可重新生成的派生产物，不得覆盖原始文件。

### 4.2 先检测，再修复

系统必须先能稳定识别问题并给出原因，再接入更复杂的解析恢复策略。不能用更换解析器掩盖缺少验收标准的问题。

### 4.3 按文档特征路由

解析策略由扫描比例、多栏、表格密度、字体编码和页面结构等特征决定，不由 QNX、Qualcomm 等厂商名称决定。

### 4.4 分层质量判断

质量判断覆盖文档、页面、结构块和 chunk。局部失败不应强制污染整份文档，也不能因为整份文档平均分合格而放过严重坏页。

### 4.5 生产门禁不可绕过

被硬门禁隔离的内容不能因 FTS、BM25 或向量命中而回到生产 Context Pack。门禁绕过只允许用于离线诊断和影子比较，并必须显式标记。

### 4.6 确定性规则优先

发布决策主要依赖确定性检测、解析器交叉验证和人工标注基准。LLM 可以辅助发现异常、解释差异或生成候选测试，但不能作为唯一验收器。

### 4.7 版本一致性

`canonical document`、chunks、FTS/BM25、向量索引和质量报告必须属于同一个不可变 release，并通过一次原子切换共同生效。

## 5. 总体架构

知识质量控制平面由六个边界清晰的组件组成。

### 5.1 Document Profiler

职责：

- 读取文件级和页面级特征。
- 识别文本型、扫描型、混合型、多栏、表格密集、代码密集等文档形态。
- 生成可版本化的 `DocumentProfile`。
- 为 Parser Router 提供路由输入。

它不负责解析正文，也不决定是否发布。

### 5.2 Parser Router

职责：

- 在代表性页面上执行低成本试解析。
- 根据画像和试解析结果选择首选解析策略。
- 将混合文档拆成逐页任务。
- 在首选策略不达标时调用备选策略。
- 记录每页选择策略的原因。

它只输出候选解析结果，不自行放行内容。

### 5.3 Quality Evaluator

职责：

- 评估文本、版面、结构、表格、证据和 chunk 质量。
- 输出原始检测信号、原因码、分项分数和建议动作。
- 区分硬失败和软降级。
- 支持新旧解析结果差异比较。

质量总分只用于排序和概览。发布决策必须同时检查硬门禁和关键分项，不能只判断平均总分。

### 5.4 Recovery Pipeline

职责：

- 根据失败原因选择替代解析器、OCR、版面增强或局部重处理。
- 限制重试次数、成本和耗时。
- 比较恢复前后的质量变化。
- 无法恢复时将内容送入隔离区。

### 5.5 Publication Gate

职责：

- 对候选 release 执行数据契约、质量、证据、检索和差异门禁。
- 决定完整发布、局部发布或拒绝发布。
- 生成可审计的发布报告。
- 确保生产检索只能读取 active release 中已放行内容。

### 5.6 Quality Registry

职责：

- 保存文件版本、解析运行、解析器配置、质量报告、release 和异常状态。
- 支持解析器升级或规则变化时的影响分析。
- 提供重处理队列和历史结果比较。
- 支持回滚和问题定位。

## 6. 文档画像与解析路由

### 6.1 DocumentProfile

画像至少包含：

- 文件格式、文件哈希、页数、加密和损坏状态。
- 每页文本字符数、文本覆盖率和图片覆盖率。
- 空白页、扫描页和混合页比例。
- OCR 平均置信度及低置信区域比例。
- 单栏、多栏和阅读顺序特征。
- 表格、代码块、列表和公式密度。
- 字体编码异常、乱码和异常字符分布。
- 页眉页脚重复度和疑似目录页范围。
- 语言组成、页面尺寸和方向变化。

画像器先对封面、目录、正文、表格密集页和尾页等代表性页面采样。若采样结果不一致，再扩展到全部页面。

### 6.2 逐页解析策略

Parser Router 支持以下策略类别：

- 原生文本提取：适用于文本层完整、版面简单的页面。
- 版面感知解析：适用于多栏、复杂标题层级和图文混排页面。
- OCR：适用于扫描页或损坏文本层。
- 混合解析：保留可信文本层，同时对图片区域或局部异常区域 OCR。
- 表格结构恢复：适用于表格密集页面，输出行、列、表头和合并关系。

系统不会对整份 PDF 固定选择一种解析器。每页可以采用不同策略，但所有页面最终必须输出统一中间模型。

### 6.3 路由与恢复状态

每页处理状态遵循：

```text
profiled
  -> primary_parsed
  -> evaluated
  -> accepted
     或 retrying
     或 quarantined
```

恢复顺序由失败原因决定，而不是固定执行所有解析器。例如：

- 文本为空或字符映射损坏：优先 OCR。
- 文本完整但阅读顺序异常：优先版面感知解析。
- 正文正常但表格结构破坏：只重处理表格区域。
- 多个解析器均无法保留证据坐标：隔离，不发布。

## 7. 多维质量模型

### 7.1 分项质量

`QualityReport` 包含以下分项：

- `text_quality`：乱码、断词、字符丢失、重复文本和语言异常。
- `layout_quality`：阅读顺序、多栏串行、文本覆盖和页眉页脚污染。
- `structure_quality`：标题层级、段落、列表、代码块和章节连续性。
- `table_quality`：表头关联、行列结构、单元格合并和数值完整性。
- `traceability_quality`：页码、坐标、原文范围、校验值和来源版本。
- `chunk_quality`：语义完整性、上下文、自包含程度和噪声比例。

每个分项保存：

- 状态。
- 分数。
- 原始指标。
- 原因码。
- 检测器及版本。
- 推荐动作。

### 7.2 发布等级

内容分为：

- A 级：关键分项和硬门禁全部通过，可正常发布。
- B 级：存在可接受的局部降级，只发布通过门禁的页面和 chunk，并携带降级标记。
- C 级：证据不可追踪、严重乱码、结构破坏或恢复失败，局部或整份隔离。

以下情况属于硬失败：

- 文档或页面应有内容但解析为空。
- 原始页数与处理页数明显不一致且无法解释。
- 已发布 chunk 无法关联到来源版本、页面和原文。
- 关键证据坐标或原文校验失败。
- 候选 release 的主数据和索引版本不一致。
- 契约引用不完整。

### 7.3 阈值管理

阈值必须：

- 通过 Golden Pages、Golden Chunks 和正常文档基准校准。
- 配置化并带版本。
- 同时包含绝对最低线和相对当前生产版本的不可回退线。
- 在改变后触发历史文档影响扫描。

## 8. 统一中间产物

流水线保存六层产物。

### 8.1 Source

保存不可变原始文件、内容哈希、发现位置和采集时间。

### 8.2 Document Manifest

保存逻辑文档身份、源版本、页数、业务元数据和 ingestion run。

### 8.3 Pages and Blocks

保存页面、标题、段落、列表、代码、表格和图片说明。每个 block 包含：

- 稳定逻辑 ID 和内容哈希。
- block 类型和父级标题路径。
- 页码、页面坐标和阅读顺序。
- 原始文本及字符范围。
- 解析器、配置和运行版本。

### 8.4 Quality Report

保存文档、页面、block 和 chunk 的分项质量、原因码和恢复历史。

### 8.5 Chunks

chunks 从 blocks 派生，不作为唯一知识事实源。每个 chunk 绑定：

- 文档和源版本。
- section path。
- 页面范围。
- block IDs 和 evidence IDs。
- chunker 版本。
- 质量等级。

### 8.6 Release Manifest

明确列出本次允许发布的：

- 文档版本。
- 页面。
- blocks。
- chunks。
- FTS/BM25 索引版本。
- 向量索引版本。
- 质量规则版本。

Layer2 只能消费 active release manifest，不再自行推断“每个文档的最新版”。

## 9. 结构感知切块

切块规则如下：

- 标题层级作为 chunk 的显式上下文。
- 段落、列表、代码块和表格尽量保持原子完整。
- 表格保存结构化表示和可检索文本表示，两者共享同一证据对象。
- 超大章节只在语义边界拆分。
- overlap 只用于同章节的跨段上下文，不复制大量正文。
- 目录、页眉页脚、版权声明和标题碎片默认标记为低检索价值。
- 每个 chunk 单独接受质量门禁。
- 邻接合并只能发生在同一来源版本和兼容结构边界内。

现有固定字符上限继续作为预算保护，但不再作为首要切分依据。

## 10. 证据链

每条检索知识必须支持：

```text
chunk_id
  -> evidence_id
  -> block/span
  -> page + bounding box + character range
  -> source_version
  -> immutable source file
```

证据对象至少包含：

- 逻辑文档 ID、标题和版本。
- 文件哈希。
- 页码和页面坐标。
- block 类型、section path 和字符范围。
- 原文和原文校验值。
- 解析器及配置版本。
- ingestion run 和 release ID。
- 质量等级。

回答层的 `evidence_ids` 只能从已检索 Context Pack 中程序化绑定。LLM 输出的未知 ID 必须丢弃，不能用于回溯。

## 11. 版本、增量与发布

### 11.1 不可变运行

每次入库创建唯一 `ingestion_run_id`：

- 相同文件哈希和相同处理配置可以复用已有产物。
- 文件变化创建新 `source_version`。
- 文件不变但解析器、chunker 或质量规则变化时，创建新的派生产物版本。
- 历史运行只读保留。

### 11.2 增量影响分析

系统根据以下变化计算重处理范围：

- 源文件变化：重处理对应文档。
- 页面级解析器变化：重处理受影响页面及其派生 chunks。
- chunker 变化：复用 pages/blocks，只重建 chunks 和索引。
- 质量规则变化：复用解析产物，重新评估并生成候选 release。
- embedding 或索引实现变化：复用 chunks，只重建对应索引。

### 11.3 原子发布

发布流程：

```text
candidate artifacts
  -> contract validation
  -> quality gates
  -> retrieval regression
  -> release manifest
  -> atomic active-release switch
```

任何步骤失败都不改变当前 active release。回滚只需把 active release 指针切换到上一已验收版本。

### 11.4 删除

删除使用 tombstone 表达。下一个 release 必须同时从主数据、FTS/BM25、向量索引和证据查询中移除对应版本，避免残留索引命中。

## 12. 自动验收与回归

### 12.1 发布前检查

候选 release 必须通过：

- 解析检查：页数、乱码、阅读顺序、结构和表格。
- 契约检查：字段、引用、版本和数据完整性。
- 证据检查：chunk 到原始文件的完整回溯。
- 检索检查：关键术语、API、精确值和问题集合的正确召回。
- 差异检查：与当前 active release 比较数量、质量和检索指标。

### 12.2 回归数据集

回归集分为：

- Golden Pages：人工标注复杂页面的文本、顺序、标题和表格结构。
- Golden Chunks：验证切块边界、上下文和证据位置。
- Golden Queries：覆盖现有 51 题、100 题和核心工程查询。
- Adversarial Queries：不存在 API、错误前提、版本冲突和精确值诱导。

QNX 和 Qualcomm 文档作为第一批复杂样本。测试分类按“扫描、多栏、表格密集、字体异常”等版面特征组织。

### 12.3 指标

至少记录：

- 已发布 chunk 的证据可追踪覆盖率。
- 页面和结构块解析成功率。
- 严重缺陷拦截率和健康样本误拦截率。
- Retrieval Recall@K。
- 正确证据进入 Top-K 的比例。
- 无有效证据回答率。
- 精确值证据支持率。
- 候选 release 相对生产版本的回退项。

### 12.4 影子检索

候选 release 可以接收生产查询副本并与当前 release 比较，但影子结果不能直接返回用户。影子检索允许查看被隔离内容用于诊断，生产检索禁止门禁绕过。

## 13. 线上监控与反馈闭环

线上记录以下质量信号：

- 检索为空或候选全部低质量。
- 回答缺少有效证据。
- 用户在短时间内重复改写相同问题。
- 同类查询的召回结果显著波动。
- 某文档、页面或解析器版本持续产生低置信结果。
- 精确值声明找不到直接支持文本。

用户反馈处理链路：

```text
feedback
  -> query/context/release association
  -> root-cause classification
  -> regression fixture
  -> generic detector or parser fix
  -> affected-corpus scan
  -> candidate reprocessing
```

只有确认问题来自单份源文件本身且无法通用处理时，才允许添加例外配置。例外必须记录原因、适用文件哈希、负责人和失效条件。

## 14. 错误处理

### 14.1 可恢复错误

依赖暂时不可用、单解析器失败、OCR 超时和局部页面异常进入有限重试。每次重试记录：

- 触发原因。
- 使用策略和配置。
- 耗时和成本。
- 恢复前后质量变化。
- 最终状态。

### 14.2 不可恢复错误

文件损坏、解密失败、证据定位不可建立或全部解析策略失败时，内容进入隔离区。隔离不影响当前生产 release。

### 14.3 系统性回退

若候选 release 出现以下情况，整次发布失败：

- 契约或索引一致性失败。
- 核心回归指标低于绝对最低线。
- 核心回归指标相对生产版本超出允许回退预算。
- 发布产物不完整。

## 15. 与现有实现的迁移关系

### 15.1 保留并演进

保留现有：

- `CanonicalDocument` 和 `Chunk` 作为演进起点。
- 稳定内容 ID 思路。
- `EvidenceSpan` 和 evidence trace。
- `parse_report` 与 PDF 文本质量 reason codes。
- pypdf 文本层优先、OCR 兜底的低成本策略。
- Layer1 到 Layer2 契约校验。
- 现有回归问题集和 Context Pack。

### 15.2 必须改变

- 将整文档 PDF 质量报告扩展到页面、block 和 chunk。
- 将 PDF 整体 OCR 兜底扩展为逐页路由和局部恢复。
- 删除生产路径中的 quality gate bypass。
- 让 Layer2 从 active release manifest 加载内容，不再自行选择最新目录。
- 将索引版本绑定到 release。
- 扩展证据对象以包含坐标、字符范围和处理版本。
- 将 QNX 独立摘要流程逐步迁移为统一流水线的输入或受控兼容适配器。

### 15.3 兼容策略

在迁移期：

- 旧 `layer1.processed.v1` 产物保持只读可消费。
- 新字段通过新 schema 版本引入，不悄悄改变 v1 语义。
- 候选 release 可同时包含旧版和新版产物，但必须在 manifest 中标明。
- 旧 QNX 数据库可继续服务当前用户，直到统一流水线通过等价或更优验收。

## 16. 分阶段落地

### 阶段 0：基线与数据契约

范围：

- 定义画像、页面、block、质量报告、运行和 release 数据契约。
- 固化源版本、解析版本和发布版本身份。
- 让主数据与索引显式绑定同一 release。
- 建立 QNX、Qualcomm 和正常 PDF 基准集。
- 记录当前质量与检索基线。

验收：

- 所有候选发布 chunk 均能追踪到源版本和页码。
- 主数据和索引版本一致性可自动验证。
- 当前生产结果可以按 release 复现。
- 基线报告可重复生成。

### 阶段 1：自动检测与发布门禁

范围：

- 实施通用质量检测器。
- 增加文档、页面、block 和 chunk 门禁。
- 建立隔离区和候选发布报告。
- 禁止生产 gate bypass。

验收：

- 基准集中已知严重缺陷全部在发布前被检测或隔离。
- 健康基准文档不被硬门禁错误拦截。
- 每次发布都有原因可审计的质量报告。

### 阶段 2：解析路由与恢复

范围：

- 接入逐页画像和试解析。
- 支持原生文本、版面感知、OCR 和表格恢复路由。
- 实施有限重试和局部隔离。

验收：

- 每页解析策略及原因可查询。
- 首选策略失败时能自动选择匹配原因的恢复策略。
- 无法修复的页面不会进入生产。
- 复杂样本质量相对基线改善，健康样本无不可接受回退。

### 阶段 3：结构感知切块与完整证据

范围：

- 按结构块重建切块策略。
- 保留表格、列表和代码的原子性。
- 完整记录坐标和字符范围。
- 程序化绑定回答证据。

验收：

- Golden Chunks 的结构边界和证据位置通过。
- 每条检索证据可以回到原始页面区域。
- 未知 evidence ID 无法进入用户答案。
- Golden Queries 的正确证据 Top-K 不低于生产基线。

### 阶段 4：增量、原子发布与回滚

范围：

- 实施不可变 ingestion run 和 release。
- 支持按变化层级计算重处理范围。
- 统一切换主数据和全部索引。
- 支持 tombstone 和回滚。

验收：

- 未变化文件不重复解析。
- 规则或组件变化只重建受影响派生层。
- 发布失败不改变生产版本。
- 回滚后主数据、索引和证据查询保持一致。

### 阶段 5：持续质量治理

范围：

- 接入影子检索和线上质量信号。
- 建立反馈归因和回归样本沉淀。
- 实施规则/解析器变更后的历史影响扫描。

验收：

- 解析器升级可自动生成受影响文档清单。
- 新质量规则可以扫描历史语料。
- 用户反馈能够关联到查询、证据和 release。
- 线上回退可定位到文档、页面、解析器和版本。

## 17. 实施优先级

实施顺序固定为：

```text
阶段 0 -> 阶段 1 -> 阶段 2 -> 阶段 3 -> 阶段 4 -> 阶段 5
```

阶段 0 和阶段 1 是体系成立的前提。没有基线和硬门禁时直接更换解析器，无法证明改善，也无法阻止新的坏结果进入生产。

阶段 2 和阶段 3 可以在接口稳定后并行准备测试样本，但生产切换仍须按验收顺序进行。

本文件是覆盖阶段 0 至阶段 5 的总体设计，不对应一个巨型实现提交。每个阶段必须作为独立交付单元：

- 单独编写实施计划。
- 拆分成可审阅的小提交或 PR。
- 完成该阶段验收后再开始下一阶段的生产切换。
- 若阶段实施暴露新的架构取舍，先补充对应阶段设计并重新确认。

设计文档通过用户审阅后，第一份实施计划仅覆盖阶段 0。阶段 1 至阶段 5 在前序阶段完成并取得实际基线数据后分别规划，避免提前固化未经验证的阈值和组件选型。

## 18. 预期优化

全部阶段完成后，预期得到：

- 从用户发现问题转为发布前主动发现问题。
- 从单一 PDF 解析策略转为按页面自适应路由。
- 从坏文档污染全库转为局部恢复和局部隔离。
- 从定长文本切片转为结构感知知识单元。
- 从页码级引用转为可核对原文区域的完整证据链。
- 从主数据与索引松散组合转为统一 release。
- 从全量重建转为按影响范围重处理。
- 从不可控发布转为候选验收、原子切换和快速回滚。
- 从厂商特例修补转为通用质量规则和特征驱动处理。

这套控制平面不会替代回答生成层 guardrail，但会显著提高进入 RAG 上下文的知识质量，为精确 API、参数、默认值和限制条件类回答提供更可靠的证据基础。

## 19. 设计决策摘要

已经确认的关键决策：

- 采用宏观质量控制平面，不采用以针对性清洗为主的方案。
- 质量控制覆盖源文件到线上反馈的完整生命周期。
- 生产低质量内容默认阻断，不允许检索命中绕过。
- 路由按文档和页面特征，不按厂商。
- 原始文件不可变，所有处理产物可重建。
- 主数据、索引和质量报告统一按 release 发布。
- 用户反馈是盲区补充，不是主要检测机制。
- 先建立基线和门禁，再增强解析器。
