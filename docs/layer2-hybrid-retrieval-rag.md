# 第二层：混合检索与 RAG 问答方案

## 一、背景

### 1.1 问题来源

第一层前处理已经解决了“资料怎么入库”的问题：QNX 官方 PDF、ClusterHMI 工程代码、配置文件、设计文档都可以被解析、清洗、分块，并保留来源信息。

但在实际问答时，我们发现仅有第一层还不够。很多失败并不是因为知识库里没有答案，而是因为**答案没有被正确检索出来**。

典型现象有两类：

- 工程师用中文描述现象，官方文档使用英文机制名。
  例如工程师会说“DMA 共享内存读到上一帧旧数据、怀疑 cache 没刷新”，而 QNX 文档里对应的是 `mmap()`、`stale data`、`PROT_NOCACHE`、`mmap_device_memory()`。
- 业务文档和官方文档混在一起后，场景词会互相干扰。
  例如“摄像头、DMA、共享内存”会命中远程哨兵/XVR 文档，也会命中 QNX 内存映射章节；“线程、CPU、UI、饿死”会命中座舱业务描述，也会命中 QNX 自适应分区调度章节。

这会导致一个很尴尬的问题：

```text
知识库里明明有答案
  -> 检索先命中了业务文档
  -> 官方机制文档排在后面或没进上下文
  -> 大模型只能基于错误证据回答
  -> 最终答成“资料里没有”
```

我们之前遇到过两个真实例子：

1. **硬件共享内存旧数据问题**
   用户描述的是“摄像头 DMA、共享内存、读到旧帧、加延时后不复现”。正确答案应该落到 QNX 的 `PROT_NOCACHE` / `mmap_device_memory()` 相关说明，但原检索先找到了 XVR 的共享内存业务文档。

2. **高优先级收包线程饿死 UI 问题**
   用户描述的是“以太网收包线程优先级太高，压力测试下 CPU 被吃满，UI 渲染线程 starvation”。正确方向是 QNX 的 Adaptive Partitioning / CPU budget / `SCHED_APS_SCHEDPOL_LIMIT_CPU_USAGE`，但原检索先被业务侧“线程、CPU、网络模块”带偏。

所以第二层要解决的不是“有没有资料”，而是：

- 用户的问题应该如何理解？
- 哪些词是现象词，哪些词对应官方机制？
- 多份文档同时命中时，应该优先相信哪类证据？
- 最终交给大模型的上下文是否足够准确、可追溯？

### 1.2 为什么要做第二层

第一层更像“资料加工厂”，它把原始材料变成可检索的 chunks；第二层更像“知识调度器”，负责从这些 chunks 里挑出真正能回答问题的证据。

如果没有第二层，系统容易退化成普通关键词搜索：

- 用户问得越长，场景词越多，越容易被业务文档带偏。
- 问题越偏底层机制，越依赖 QNX 官方术语桥接。
- 大模型上下文越杂，越容易给出看似合理但证据不对的回答。

因此第二层的核心价值是：

```text
把“用户自然语言问题”
转成“正确的检索意图 + 官方机制词 + 可追溯证据包”
再交给大模型回答。
```

它不是替代第一层，而是在第一层之上增加检索判断、证据排序和回答约束。

### 1.3 目标与定位

第二层混合检索与 RAG 问答方案的目标是：

1. **一个入口查所有知识**
   用户不需要手动区分“我要查 QNX”还是“我要查远程哨兵”。系统默认在统一知识库里检索，再自动选择最相关证据。

2. **让中文问题能命中英文机制**
   对 QNX/嵌入式常见问题做术语桥接，例如 cache stale data、Adaptive Partitioning、Resource Manager、IPC 等。

3. **减少业务文档干扰**
   当官方机制文档和业务文档都命中时，优先把能解释底层机制的证据排到前面。

4. **给大模型可控上下文**
   不把原始文件直接塞给 LLM，而是提供 Context Pack：包含原文片段、来源文档、页码、章节、得分、检索信号和证据 ID。

5. **本地可部署、可离线运行**
   核心检索依赖 SQLite FTS/BM25；BGE-M3 可选，用于增强语义召回，但不是强依赖。

## 二、技术模块

### 2.1 端到端数据流

暂时无法在飞书文档外展示此内容

文字版流程如下：

```text
第一层知识库
  -> 用户问题
  -> 查询理解与术语扩展
  -> BM25 / FTS / 向量多路召回
  -> 候选证据融合排序
  -> Context Pack 证据包
  -> LLM 基于证据回答
  -> 飞书 / API / 本地 Agent 输出
```

第二层不重新解析 PDF，也不重新扫描源码；它只消费第一层已经处理好的结构化结果。

### 2.2 查询理解：从“现象描述”到“机制词”

工程师提问通常不是官方文档式表达，而是排障现场语言。第二层会先判断问题里是否包含某类底层机制线索，再补充对应官方术语。

问题现象 | 系统补充的机制词 | 目标文档方向
---|---|---
DMA 写共享内存，读到旧帧，怀疑 cache | `mmap`、`stale data`、`PROT_NOCACHE`、`MAP_PHYS`、`mmap_device_memory` | QNX System Architecture / C Library Reference
高优线程吃满 CPU，UI 被饿死，希望自动限流 | `Adaptive Partitioning`、`CPU budget`、`over budget`、`SCHED_APS_SCHEDPOL_LIMIT_CPU_USAGE`、`max_budget_percent` | QNX Adaptive Partitioning User's Guide
资源管理器、驱动、设备节点、read/write/devctl | `Resource Manager`、`resmgr`、`devctl`、`iofunc` | QNX Resource Manager / C Library
进程间通信、消息、channel、pulse | `MsgSend`、`ChannelCreate`、`ConnectAttach`、`pulse` | QNX IPC / C Library

这一步不是让系统“编答案”，只是让检索知道应该去找哪些官方术语。

### 2.3 多路召回：为什么不能只靠一种检索

第二层同时保留多种召回方式，是因为不同问题适合不同技术。

检索方式 | 适合什么 | 为什么需要它
---|---|---
BM25 / FTS | API、宏、函数名、错误码、英文术语 | 对 `PROT_NOCACHE`、`MsgSend()` 这类精确符号最可靠
中文 lexical | 用户问题里的场景词 | 能保留“摄像头、线程、UI、报文”等工程语境
BGE-M3 向量 | 中文现象与英文概念之间的语义匹配 | 能降低“说法不同导致搜不到”的概率
技术术语扩展 | QNX/嵌入式高频坑位 | 把经验沉淀成稳定召回规则
元数据/质量过滤 | 文档版本、项目、解析质量 | 保证来源可靠、结果可解释

简单说：

```text
BM25/FTS 负责精确命中
BGE-M3 负责语义接近
规则扩展负责领域经验
Context Pack 负责证据可控
```

### 2.4 证据融合：避免“找到了但没排上来”

多路召回后，第二层会对候选进行融合排序。排序时不仅看文本相似度，还会看：

- 是否命中官方机制词；
- 是否来自更可靠的文档片段；
- 是否覆盖用户问题中的关键子句；
- 是否与主题一致；
- 是否有 FTS/vector 明确信号；
- 是否能通过 evidence id 追溯到原文。

这里有一个重要细节：第一层可能会生成原始 chunk，也可能生成相邻 chunk 合并后的 chunk。外部索引有时命中原始 chunk，但最终证据包更适合选合并 chunk。第二层会通过 `evidence_id` 把命中分数传播过去，避免出现：

```text
FTS 找到了正确原文
  -> 但最终 selected chunk 是另一个合并块
  -> 分数没有传过去
  -> 正确证据没进 Context Pack
```

这也是近期修复 QNX cache / APS 问题时最关键的一点。

### 2.5 Context Pack：给大模型的“证据包”

第二层最终不会把一堆散乱文本直接交给大模型，而是输出 Context Pack。

Context Pack 包含：

内容 | 作用
---|---
问题原文 | 保留用户真实意图
选中的证据片段 | 给 LLM 的主要参考资料
文档标题和版本 | 判断资料来源
页码和章节 | 方便工程师回查
综合得分 | 表示排序结果
检索信号 | 说明是 BM25、FTS、vector 还是 topic 命中
evidence id | 支持追溯到原始 evidence span
质量状态 | 避免低质量解析结果污染回答

它的作用是把回答边界框住：

```text
LLM 只能基于 Context Pack 回答；
如果 Context Pack 没有证据，就应该说资料不足，不能编。
```

### 2.6 飞书问答接入

飞书 bot 是第二层的一个入口，不是独立检索系统。

消息处理流程：

```text
用户在飞书提问
  -> bot 收到消息
  -> 调用本地 API 获取 Context Pack
  -> 将 Context Pack 格式化为参考资料
  -> LLM 基于参考资料合成回答
  -> bot 回复飞书
```

这样做的好处是：

- 飞书 bot 不重复实现检索逻辑；
- API、CLI、本地 Agent 可以复用同一套检索结果；
- 修复一次检索策略，所有入口都会受益；
- 未来替换 LLM 或增加 BGE-M3，不影响飞书接入方式。

### 2.7 BGE-M3 的定位

BGE-M3 后续可以作为第二层的语义增强模块，但它不是替代 BM25/FTS 的东西。

能力 | BM25/FTS | BGE-M3
---|---|---
命中 API/宏/函数名 | 强 | 中
中文问题匹配英文概念 | 弱，需要规则辅助 | 强
离线部署成本 | 低 | 需要模型文件
构建速度 | 快 | CPU 慢，建议 GPU 构建
可解释性 | 高 | 中

推荐组合：

```text
BM25/FTS 精确召回
  + BGE-M3 语义召回
  + QNX 术语扩展
  + 证据融合排序
```

这样既能处理 `PROT_NOCACHE` 这种符号级问题，也能处理“旧数据”“饿死线程”这种自然语言问题。

## 三、总结

第二层混合检索与 RAG 问答流程，是第一层前处理之后的“证据选择与回答控制层”。

第一层解决：

- 原始资料如何解析；
- 如何分块；
- 如何入库；
- 如何保留来源。

第二层解决：

- 用户问题如何理解；
- 中文现象如何桥接到 QNX 官方术语；
- 多份文档同时命中时如何排序；
- 如何避免业务文档压过官方机制文档；
- 如何把证据组织成大模型可用、工程师可复核的 Context Pack。

核心取舍：

- 保留 BM25/FTS，因为符号级 API、宏、flag 需要精确命中。
- 引入 BGE-M3 作为可选增强，用于跨语言、跨表达的语义召回。
- 保留 QNX 领域术语扩展，因为很多底层问题靠通用语义模型也未必稳定。
- 用 Context Pack 限制 LLM 的回答范围，避免“看似懂了但证据错了”。

当前第二层已经能覆盖两类典型底层问题：

- QNX 内存映射 / cache stale data / `PROT_NOCACHE`。
- 高优线程 CPU starvation / Adaptive Partitioning / CPU budget。

后续如果继续扩展，应优先沉淀更多 QNX 高频问题的术语桥接，例如 Resource Manager、IPC、SMMU、Hypervisor、Interrupt、Timer、Memory Protection 等。这样系统会越来越像“懂 QNX 开发语境的检索助手”，而不是普通全文搜索。
